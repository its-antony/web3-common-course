# Part07 - Web3 项目快速判断方法

## ADDED Requirements

### Requirement: 7.1 普通人为什么不该"找好项目"

本章节 SHALL 帮助学员理解普通人在项目筛选上的局限性，建立正确的参与心态。

内容 MUST 包含：
- 信息不对称问题的解释
- 时间成本与机会成本分析
- 与专业投资者的能力差距
- 正确的参与心态建议

#### Scenario: 学员理解信息不对称

- **WHEN** 学员阅读本章节
- **THEN** 能够用一句话解释为什么普通人在项目筛选上处于劣势

#### Scenario: 学员建立正确心态

- **WHEN** 学员完成本章节学习
- **THEN** 能够描述普通人参与 Web3 时应持有的 3 个正确心态

---

### Requirement: 7.2 项目快速扫描的核心维度

本章节 SHALL 提供一个实用的项目评估框架，帮助学员快速扫描项目基本面。

内容 MUST 包含以下评估维度：
- 团队背景（公开度、过往项目、行业声誉）
- 融资情况（投资机构、融资金额、融资轮次）
- 技术可行性（白皮书质量、代码仓库、审计报告）
- Token 经济模型（分配比例、解锁计划、通胀率）
- 社区活跃度（真实用户、互动质量、机器人识别）
- 产品进度（路线图、实际交付、版本迭代）

内容 SHOULD 包含：
- 每个维度的具体检查方法
- 常用数据查询工具和平台
- 至少一个真实项目的扫描案例

#### Scenario: 学员掌握评估框架

- **WHEN** 学员完成本章节学习
- **THEN** 能够使用 6 个核心维度对任意项目进行初步评估

#### Scenario: 学员能够查询项目信息

- **WHEN** 学员需要查询某个项目的融资信息
- **THEN** 知道至少 2 个可用的数据平台（如 RootData、Crunchbase）

---

### Requirement: 7.3 什么时候可以直接不看一个项目

本章节 SHALL 提供一份「红旗信号」清单，帮助学员快速排除高风险项目。

内容 MUST 包含以下红旗信号：
- 匿名团队 + 高收益承诺组合
- 代码未开源且无第三方审计
- Token 分配高度集中（团队/早期投资者占比过高）
- 社区充斥明显的机器人活动
- 白皮书空洞、无技术细节
- 已有负面新闻或团队黑历史

内容 SHOULD 包含：
- 每个红旗信号的具体案例说明
- 如何快速核实这些信息
- 「一票否决」与「综合判断」的区分

#### Scenario: 学员识别高风险项目

- **WHEN** 学员遇到一个项目同时满足「匿名团队」和「高收益承诺」
- **THEN** 能够判断这是一个红旗信号，应该直接跳过

#### Scenario: 学员理解红旗信号的权重

- **WHEN** 学员完成本章节学习
- **THEN** 能够区分哪些红旗信号是「一票否决」型，哪些需要综合判断

---

### Requirement: 7.4 信息源与噪音的区分方法

本章节 SHALL 帮助学员建立有效的信息筛选系统，区分高质量信息源和噪音。

内容 MUST 包含：
- 一级信息源定义与示例：官方 Twitter、Discord、Telegram、官网公告
- 二级信息源定义与示例：媒体报道、KOL 分析、研究报告
- 数据平台介绍：RootData、DefiLlama、Dune、Token Terminal、Nansen、Arkham
- 付费软文和虚假信息的识别方法
- 建立个人信息筛选系统的建议

内容 SHOULD 包含：
- 推荐关注的高质量账号/媒体清单
- 英文 vs 中文信息源的对比
- 信息验证的 DYOR 方法论

#### Scenario: 学员区分信息源层级

- **WHEN** 学员需要了解某个项目的最新进展
- **THEN** 知道应该优先查看一级信息源（官方渠道）

#### Scenario: 学员识别付费软文

- **WHEN** 学员阅读一篇关于某项目的「深度分析」
- **THEN** 能够通过至少 3 个特征判断这是否为付费软文

#### Scenario: 学员建立信息系统

- **WHEN** 学员完成本章节学习
- **THEN** 能够在 X 上关注 5-10 个高质量信息源，并说明选择理由

### Requirement: Part07 学习任务

本 Part MUST 包含可验证的学习任务，帮助学员将知识转化为实践。

主要学习任务 SHALL 包含：
- 在 X 上关注 5-10 个高质量账号/媒体/项目方，列出关注清单并说明理由

可选学习任务 SHOULD 包含：
- 选择一个热门项目，使用 7.2 的框架进行完整扫描并输出评估报告
- 找出一篇疑似付费软文，分析其特征

#### Scenario: 学员完成必做任务

- **WHEN** 学员完成 Part07 学习
- **THEN** 能够提交一份包含 5-10 个高质量信息源的关注清单

---

### Requirement: Part07 文档规范

所有 Part07 的课程文档 MUST 遵循「课程文档编写规范」。

每篇文档 MUST 包含：
- 学习目标（1-3 条，使用「能够……」句式）
- 核心内容（使用三级标题组织）
- 案例/故事（200-500 字真实案例）
- 关键概念速查（表格形式，3-8 个概念）
- 学习资料（必读 1-3 个 + 选读 2-5 个）
- 学习任务（1-3 个可验证任务）
- 常见问题 FAQ（2-5 个问题）
- 版本信息（更新日期、编写人、审核人）

每篇文档 SHOULD 遵循：
- 中英混排原则
- 专业术语首次出现写全称
- 每段不超过 5 行
- 关键数据标注来源

#### Scenario: 文档通过规范检查

- **WHEN** 审核人检查 Part07 的任意文档
- **THEN** 文档包含规范要求的所有必备模块

#### Scenario: 资料引用符合规范

- **WHEN** 文档引用外部资料
- **THEN** 必读资料来自 S/A 级来源，所有数据标注来源和时间
